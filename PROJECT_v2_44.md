## 七、抽獎系統

### 核心原則
無論哪個活動模組，持幣量永遠有加權。

### 最終權重公式
```
最終權重 = 基礎票數 × 持幣排名加權 × 時間加權 × 鐵粉加權
```

**注意：卡牌不列入權重公式。權重卡效果直接疊加在持幣排名加權上，獎金卡影響中獎後的獎金金額，兩者皆不改變公式結構。模組只決定參加資格，不影響權重計算。**
**虧損補貼加權（×1.5）為不定期特別福利，特殊活動才加入，不列入常規公式。**

---

### 鐵粉加權（每日0.1U超級鐵粉計畫）

**計算方式：線性累積，月底清空**
- 每天購買一次 0.1U JIUCAI，當月鐵粉權重 +1%
- 最高累積 31%（連續整月）
- 每月 1 號自動清空，重新開始
- 部分天數參與也有效果（非全有全無設計）

**鐵粉加權公式：**
```
鐵粉加權 = 1 + (本月累積天數 × 0.01)
例：買了10天 → ×1.10；買了31天 → ×1.31
```

**個人資料頁新增顯示：**
```
本月鐵粉權重：+18%（本月已買18天）[每日0.1U 按鈕]
```

**按鈕行為：**
- 按下「每日0.1U」自動帶出購買 0.1U JIUCAI 的錢包確認提示
- 用戶確認後完成購買，系統記錄當日已執行
- 每日限一次，當天已購買則按鈕顯示「今日已完成 ✅」

**首波同步上線。**

### 卡牌加權

| 卡牌類型 | 影響環節 | 說明 |
|---------|---------|------|
| 權重翻倍卡 | 抽中機率 | 提升權重，讓你更容易抽中，有不同倍數版本 |
| 獎金翻倍卡 | 中獎金額 | 抽中後獎金倍增，不影響權重，有不同倍數版本 |
| 無敵星星卡 | 資格卡 | 持有期間自動享當期最高排名權重，不佔用張數名額 |

**使用規則：**
- 權重翻倍卡＋獎金翻倍卡**合計最多使用 2 張**，可混搭（例：權重卡1張＋獎金卡1張、或權重卡2張、或獎金卡2張）
- 疊加採乘法邏輯：權重+2x × 2張 = 2x × 2x = ×4
- 無敵星星卡為限期生效，到期自動失效，不消耗，**不佔2張名額**
- 未來可擴充新卡片，不影響現有系統

### 權重計算範例（必須展示在抽獎說明區）

**用戶A（老韭菜）：**
- 持幣排名：前20%（3.2x）
- 持倉時間：45天（1.5x）
- 持有權重+2x卡 2張（2x × 2x = 4x，卡牌疊加為乘法邏輯）
- 本月鐵粉天數：15天（×1.15）

```

## 十五、卡牌系統

**核心精神：讓每次參與都有收穫，沒中獎也不空手而回。卡牌可狂灑，是重要激勵工具，讓後段班有逆襲機會。**

**現行卡牌種類（簡化版）：**

| 卡牌 | 類型 | 效果 | 備註 |
|------|------|------|------|
| 權重翻倍卡 | 權重卡 | 提升抽中機率，有 +1.5x / +2x 等不同版本 | 最多使用2張，乘法疊加 |
| 獎金翻倍卡 | 獎金卡 | 中獎後獎金倍增，有 +2x / +3x 等不同版本 | 最多使用2張，乘法疊加 |
| 無敵星星卡 | 資格卡 | 持有期間自動享當期最高排名權重 | 不佔2張名額，限期生效 |
| 獲利了結卡 | 特殊卡 | 活動期間部分減持可獲得，補足因減持損失的權重 | 不佔2張名額，細節待定 |

**獲利了結卡設計概念：**

核心理念：鼓勵用戶在幣價上漲時部分獲利了結，製造交易量與稅收，同時維持社群活躍度。

```
觸發條件：幣價漲超過100%時啟動活動
追蹤期間：活動開始後兩週
減持比例與獎勵：
  減少 1–15%  → 獲得 ×1.15 權重卡
  減少 15–20% → 獲得 ×1.2 權重卡
  減少超過 20% → 不給卡
  未減持 → 不給卡（此卡專為獎勵獲利了結設計）
```

**推播文案：**
> 「$JIUCAI 已漲超100%，你的持倉現在值 $XXX。先落袋一部分？幣還在，抽獎資格還在，跌了再補，又是一次機會。」

**設計優勢：**
- 部分賣出時間權重不歸零（未清空持倉）
- 用戶實現真實獲利，建立對項目的信任
- 賣出＋日後買回 = 雙向交易量 = 稅收持續進來
- 不會清空流動池，因為只是部分了結

**細節待定：** 卡的有效期、計算基準、活動頻率，上線後視情況規劃。

**疊加規則：**
- 權重卡與獎金卡**合計最多 2 張**，可混搭（例：權重卡1張＋獎金卡1張）
- 乘法邏輯：權重+2x × 2張 = 2x × 2x = ×4
- 無敵星星卡到期自動失效，不需手動消耗

**擴充原則：**
未來新增卡片直接告知 Claude Code，設計邏輯、資料庫、前端一起處理，不影響現有系統。

**卡牌取得方式：**
- 每期抽獎部分送幣、部分送卡牌
- 特殊活動發放
- 推薦朋友隨機獲得

**啟用機制：**
用戶在個人資料頁設定每張卡牌啟用數量，系統在下次抽獎自動套用後消耗。
抽獎行事曆公開，用戶看情況在抽獎前決定啟用數量。

**顯示格式：**
```
🃏 權重+2x卡 x3  [本次抽獎：啟用 2 張 ▼]
⭐ 無敵星星卡 x1  [生效中：還有18天]
🃏 獎金+3x卡 x1  [本次抽獎：啟用 ✅]
```

```

swap 比例（N%）存於環境變數，隨時可調整。

   - **確認** → 系統立刻執行打幣和推播

**細節修改需求透過 Claude Code 討論處理。**

**技術實現：**
- 所有打幣統一使用批次轉帳合約，無論人數多少
- 一筆交易同時打到所有得獎地址，Gas 費只付一次
- 10人批次費用約 $0.05–0.1，1000人批次費用約 $2–5

## 四十六、資料庫擴充性規劃

- 並發連接數過高 → 提醒加快取

**停機最小化原則：**
- 所有資料庫結構變更使用 Migration 腳本
- 重要操作前先備份
- 升級在低流量時段（凌晨）執行

## 三十一、推播機制

**核心目的：讓每次開獎成為真實事件，建立長期信任感**

中獎通知僅針對中獎人，沒中的的不要推播，但抽獎預告全員推播。

**推播短句確認機制：**

每次批次打幣前，Claude 準備以下內容供創辦人確認：
1. 得獎名單
2. 獎金金額
3. 本次推播短句

確認後才發送，確保每次推播都有人味，不是冷冰冰的系統訊息。

**短句風格（每次擇一，創辦人可修改）：**

鼓勵型：
- 「繼續撐著，韭菜不倒。」
- 「沒中？下次卡牌用起來。」

預告型：
- 「下週大獎，獎池已累積 $XXX，別賣。」
- 「本週五最慘局開抽，虧越多越有機會。」

梗型：
- 「又送出去了，交易所你們看著。」
- 「恭喜中獎，但別忘了繼續虧。」

---

**推播管道（雙軌並行）：**
- **Telegram Bot（主力）**：用戶加入 Bot 後自動接收通知，中文幣圈接受度最高
- **Web Push（補充）**：網站連錢包時請求瀏覽器通知權限，覆蓋不用 Telegram 的用戶

**Telegram 三頻道架構（完全隔離）：**

| 頻道 | 用途 | 收訊對象 |
|------|------|---------|
| 創辦人私人頻道 | 系統日報、異常警報、開獎確認按鈕、獎池警戒 | 只有創辦人 |
| JIUCAI_bot（會員用）| 開獎通知、中獎推播、活動預告群發、會員私訊 | 會員 + 會員私訊時通知創辦人 |
| Jiucai 公開頻道 | 對外公告 | 所有訂閱者 |

**推播流向明細：**
- 系統相關推播 → 創辦人私人頻道
- Bot 對會員的群發推播 → 只進 JIUCAI_bot，不通知創辦人
- 會員主動私訊 Bot → 進 JIUCAI_bot + 另外通知創辦人
- 對外公告 → Jiucai 公開頻道，手動或 Claude Code 協助發布

**三個頻道完全隔離，系統推播不混進會員 Bot，會員群發不打擾創辦人。**

**JIUCAI Bot 會員詢問處理流程：**

會員傳訊息給 Bot → Bot 推播通知創辦人手機 → 創辦人在 Telegram 直接回覆 → Bot 轉發給會員。會員全程只看到 Bot，不知道後面是創辦人在回。

| 問題類型 | 處理方式 |
|---------|---------|
| 常見問題、規則查詢 | 網站已有完整說明，Bot 引導至網站 |
| 爭議、疑問 | 推播通知創辦人，創辦人親自判斷回覆 |
| 創辦人回覆 | 直接在 Telegram 打字，或告知 Claude Code 起草後發出 |

**原則：** 公開疑問引導至「我有話要說」區塊讓社群自行討論，官方選擇性置頂回覆。私下爭議才走 Bot，降低創辦人處理量。

**Bot 訊息過濾機制（預設關閉，視上線後狀況開啟）：**

Bot 收到訊息後先判斷類型，再決定是否推播給創辦人：
```
訊息包含關鍵字（退款、騙、詐騙、客服、投訴、沒收到等）
→ 推播通知創辦人

其他一般訊息
→ Bot 自動回覆引導至網站或 Telegram 群，不打擾創辦人
```
關鍵字清單存於環境變數，隨時可調整新增。預設關閉，所有訊息直接推播，待訊息量大後再視情況開啟過濾。

**強制推播時機：**
- 開獎結果（含得獎地址、獎金金額、交易 hash）
- 個人中獎通知
- 獎池金額突破新高
- 新活動公告

**選擇性推播：**
- 每週憤怒值排行更新
- 幣價重大變動

---

**開獎節奏（減低賣壓核心機制）：**

持幣才有資格 → 賣了失去資格 → 捨不得賣 → 賣壓自然降低

**核心邏輯：先累積相信的人，獎金大小是其次**
每天多次小額真實中獎 → 一週累積數十個真實得獎者 → 基本盤自然形成 → 獎金再慢慢增加

| 層級 | 頻率 | 獎金類型 | 金額 | 目的 |
|------|------|---------|------|------|
| 日常小獎 | 一天多次（可達三次） | USDT | $1–5/人 | 建立真實感和信任 |
| 週度中獎 | 每週一次 | USDT + 韭菜幣 | 中等 | 雙重獎勵，維持期待 |
| 月度大獎 | 每月一次 | 主要USDT + 大量韭菜幣加碼 | 最大 | 重點行銷時機 |
| 特殊臨時獎 | 不定期 | 視情況 | 視情況 | 幣價下跌時穩住信心 |

**推播原則：**
- 日常小獎推播簡短：「剛剛又有人中獎了 💚」
- 週度/月度大獎完整說明：得獎地址、金額、交易hash
- 避免完整推播過於頻繁導致用戶關閉通知

**初期成本估算：**
一天10人各 $1 USDT = $10/天 = $300/月
金額小但累積信任效果遠超成本。

**獎金幣種策略：**
- USDT：主力，感受直接，初期建立信任必用
- 韭菜幣：加碼用，成本低，適合放大噱頭
- 原則：USDT 建信任，韭菜幣放大獎勵感受

## 五十三、資金安全防護機制

**最高風險點：Railway 環境變數洩漏導致操作錢包被清空。**

### 三層防護

**第一層：打幣白名單（資料庫已知地址）**
系統打幣的對象永遠只有資料庫裡連結過的錢包地址。程式層面拒絕對資料庫以外的陌生地址發起任何打幣交易。就算私鑰洩漏，攻擊者必須先把目標地址塞進資料庫才能收款。

**第二層：活動時間軸鎖定（異常偵測核心）**
每次創辦人告知 Claude Code 新活動後，系統自動排定完整時間軸：
- 預告推播時間
- 資格掃描時間
- 開獎時間
- 打幣時間

**時間軸以外的任何出帳動作 = 異常，立即推播 Telegram 警報。**

觸發異常警報的條件：
- 非排程時間點的打幣
- 收款地址不在資料庫
- 單筆金額超過上限

**第三層：單筆打幣上限**
每筆交易最多打出 $1,500 USDT（預設值，存於環境變數可調整）。超過上限自動分批並觸發人工確認。防止私鑰洩漏時被單筆清空。

### 合約層補充防護（活動合約）
參考「金庫／邏輯分層」原則：活動合約每日可提款金額設上限，就算合約邏輯被攻擊，單日損失有限。具體上限視活動規模由 Claude Code 建議後創辦人確認。

### 開發實作要求（Claude Code 必須實作，非選配）

**1. 打幣前地址驗證**
```
每次打幣前強制查詢：
收款地址是否存在於 wallets 資料表？
→ 否：拒絕交易，記錄 log，推播 Telegram 警報
→ 是：繼續執行
```

**2. 時間軸監控守護程序**
```
常駐 Railway 背景執行，每分鐘掃描一次：
是否有打幣交易發生在排程時間點以外？
→ 是：立即推播 Telegram 警報，附上交易 hash、金額、收款地址
→ 否：正常，不動作
```

**3. 單筆金額上限檢查**
```
每次打幣前檢查：
單筆金額是否超過環境變數 MAX_SINGLE_PAYOUT（預設 1500 USDT）？
→ 超過：自動拆分為多筆 + 推播確認通知創辦人
→ 未超過：直接執行
```

**4. 異常警報格式（Telegram）**
```
⚠️ 資金異常警報
類型：[非排程打幣／陌生地址／超額]
時間：2026-04-13 03:22
金額：$XXX USDT
收款地址：0x...
交易Hash：0x...（如已發出）
→ 請立即上 Claude Code 確認
```

### 異常發生時的處理流程

**情況一：打幣前被攔截（交易尚未發出）**
```
系統發現異常 → 拒絕執行交易 → 記錄 log
→ 推播 Telegram 警報
→ 創辦人上 Claude Code 確認原因後手動解鎖
不需要暫停整個系統
```

**情況二：打幣已出去才偵測到異常**
```
1. 系統立即暫停所有排程打幣
2. 推播 Telegram 警報 + 兩個按鈕：
   🔴 緊急暫停 → 停止所有後續打幣排程
   ✅ 確認正常 → 恢復系統繼續執行
3. 創辦人按「緊急暫停」後系統進入人工確認模式
4. 人工確認模式下所有打幣需創辦人手動解鎖才能執行
5. 創辦人上 Claude Code 確認：誰、多少錢、怎麼發生
6. 確認安全後手動恢復系統
```

**開發實作要求（Claude Code 必須實作）：**
- 系統狀態欄位 `system_status`：`normal` / `suspended`
- `suspended` 狀態下所有打幣函式強制返回，不執行
- Telegram 緊急暫停按鈕直接寫入 `system_status = suspended`
- 恢復需創辦人在 Claude Code 明確下指令，不可自動恢復



詳細五個錢包架構與自動化流程見第十七章獎池結構。

**核心原則：**
- Owner 錢包採用 Gnosis Safe 多簽（1-of-2），兩個地址均為創辦人持有（手機A + 手機B），任一簽名即可執行，互為備援
- 個人錢包私鑰永遠只在創辦人手上
- 需要自動化的三個操作錢包私鑰放 Railway 環境變數
- 個人錢包只收系統轉入的乾淨資產，不直接參與任何自動化操作

**Owner 錢包操作方式：**
透過 safe.global 介面操作，連接其中一個錢包簽名即可執行。與普通錢包操作差不多，只是多了 Safe 介面一層。使用頻率極低，僅在部署新合約或調整合約設定時使用。

**核心原則：核心合約穩定不動，活動合約獨立部署可替換。**

### 合約拆分

| 合約 | 功能 | 是否Renounce |
|------|------|-------------|
| 核心代幣合約 | 發行、稅率、轉帳 | 是（發行後鎖死）|
| 抽獎合約 | 加權抽籤、打幣 | 否（需更新活動參數）|
| 預測市場合約 | 押注、結算、銷毀 | 否 |
| 天使席位競標合約 | 收標金、加權抽籤、銷毀標金 | 否 |

### 同一區塊交易限制（防機器人攻擊）

合約記錄每個地址最後交易的區塊號，同一地址在同一區塊內只能交易一次，下次交易需等下一個區塊（約2秒）。

- 防止機器人在極短時間內大量買賣砸盤
- 普通用戶完全感覺不到，不影響正常交易體驗
- 屬於合約層硬性限制，無法繞過
- **不對外公告**，讓攻擊者自己去撞

核心代幣合約 Renounce 後，社群對主幣完全信任。活動合約各自獨立，壞了或需升級直接換新合約，不影響主幣。

### 免稅白名單

系統營運錢包設為免稅白名單，內部轉帳和打幣不被扣交易稅：

| 錢包 | 原因 |
|------|------|
| Owner 錢包 | 合約操作免稅 |
| JIUCAI 獎池錢包 | 打獎金給會員不被扣稅 |
| USDT 獎金錢包 | 打 USDT 獎金不被扣稅 |
| 運營中轉錢包 | 內部調撥不被扣稅 |

**個人錢包不列入白名單**——創辦人個人買賣照樣收稅，公平對待。

**白名單管理權限在設定完四個錢包後一併放棄（Renounce）**，之後無法新增任何地址，社群可鏈上驗證，無後門疑慮。

### 天使席位競標合約邏輯

1. 用戶將任意數量 JIUCAI 打入競標合約
2. 活動截止，合約依投標金額加權抽籤
3. 抽中者取得天使席位資格，享當月分紅
4. 所有標金自動銷毀（送黑洞地址），無人可取回

標金銷毀為通縮機制，同時解決「創辦人是否吃標金」的信任疑慮。

### 抽獎資格篩選：後端邏輯，非合約

複雜的資格條件（如「持倉虧損30%以上」、「持倉超過60天」）由 Claude Code 查詢資料庫與鏈上數據產生符合資格地址名單，再將名單與權重傳入抽獎合約執行。

**好處：** 資格條件隨時可改，不受合約限制。合約只做最單純的事——收到名單＋權重 → 抽籤 → 打幣。

## 五十、合約安全聲明（韭菜宣言用）

**關於合約安全**

- **鑄幣權已放棄（Renounced）**：$JIUCAI 發行後創辦人永久放棄增發能力，總量 100 億顆，鏈上可查，任何人包括創辦人都無法再鑄造新幣。所謂的「動態補充機制」，是創辦人將自己持有的份額釋出，而非憑空增發。
- **稅率上限寫死合約**：買賣稅率總計永不超過 6%，合約層面硬性限制，無法調高。稅收內部分配比例（獎池／流動池／運營）可視項目階段調整，任何調整提前 48 小時公告，鏈上可查。
- **重大操作設有 48 小時 Timelock**：任何合約層級的變更，社群有 48 小時提前知情，鏈上公開透明。
- **合約地址公開**：部署後完整地址公告，任何人可在 Basescan 自行查閱所有操作紀錄。
- **部署前自動化安全掃描**：合約由 Claude Code 撰寫完成後，自動執行 Slither 掃描，修復所有發現的已知漏洞後才部署，掃描報告留存備查。
- **尚未第三方審計**：啟動資金優先投入獎池與社群。計畫在交易量穩定後安排 SolidProof 或同級機構審計並公開報告。

**最強的保障不是報告，是每一筆打出去的獎金——全部鏈上，永遠可查。**

### 開發實作要求（Claude Code 必須實作，非選配）

```
每份合約部署前強制執行：
1. 安裝 Slither（pip install slither-analyzer）
2. 對合約跑完整掃描：slither contracts/Token.sol
3. 讀取報告，列出所有 High / Medium 風險項目
4. 逐一修復，修復後重新掃描確認清零
5. 回報創辦人：「掃描完成，發現 X 個問題，已全部修復」
6. 保存最終掃描報告至 /reports/slither_[合約名稱].txt
確認無誤後才進行部署。
```

## 五十五、抽獎計算錯誤處理原則

**風險評估：** 可能發生但不會嚴重，上線前以測試資料驗證可抓掉大部分問題。常見錯誤類型：排名邊界值計算錯誤、鐵粉天數跨月清零時機、卡牌乘法順序。

**核心原則：打出去的不追，用下次補。**

鏈上交易不可逆，追回獎金技術不可能，強行追回是信任災難。

**分情況處理：**

| 情況 | 處理方式 |
|------|---------|
| 誤差小，社群未察覺 | 靜默修正邏輯，下次生效，不公告 |
| 明顯不公平（如某人權重暴增）| 主動公告說明，對受影響者補發差額，已發出的不追回 |
| 系統性大量誤發 | 暫停下次開獎，公告說明修復過程，重開一次補償抽獎 |

**公告範本（需要時使用）：**
> 「我們發現本次抽獎權重計算存在問題，已修正。受影響的用戶將獲得額外補發，感謝社群的理解與支持。」

**開發實作要求（Claude Code 必須實作）：**
```
每次開獎前：
1. 輸出前10名權重最高的地址與對應數值供人工抽查
2. 檢查是否有異常高權重（超過理論最大值）
3. 有異常 → 推播警報，暫停開獎等待確認
```

## 五十四、自主模式與知識庫

**核心理念：最差的情況是不再創新，但系統能持續維持獎金送出、社群不死。**

---

### 自主模式觸發條件

部署心跳機制於 Railway，記錄創辦人最後回應時間。
超過 **14天** 未收到心跳確認 → 系統自動切換進自主模式。

心跳機制細節待上線後設計，預計為每週一次簡單 Telegram 回覆確認。

---

### 自主模式決策框架

**獎池監控與開獎頻率：**
```
獎池充足（> 警戒線 × 3）→ 維持正常頻率，可小幅加碼
獎池普通（警戒線 × 1–3）→ 正常頻率，標準獎金
獎池偏低（< 警戒線）    → 降頻，小額維持存在感
獎池極低（< 最低門檻）  → 暫停開獎，推播說明，等補充
```

警戒線與最低門檻數值由創辦人設定於環境變數，上線後根據實際情況校準。

**自動決策邏輯：**
```
每週執行一次：
1. 查詢獎池餘額
2. 估算本週預期流入（根據近7日平均交易量）
3. 根據上方框架決定本週開獎次數與金額
4. 按歷史模式設定資格條件（持倉天數、排名門檻）
5. 大小獎比例自動分配（參考歷史開獎紀錄）
6. 排定時間軸，自動執行打幣、推播、更新紀錄
```

**自主模式不處理的事：**
- 新功能開發
- 合約層級操作
- 危機應對（系統異常、攻擊事件）
- 上述情況發生時：暫停系統，持續推播等待人工恢復

---

### 知識庫建立原則

Claude Code 沒有持久記憶，但可透過持續累積文件達到類似學習的效果。

**知識庫組成：**

| 文件 | 內容 | 更新時機 |
|------|------|---------|
| PROJECT.md | 完整架構與設計決策 | 每次重大決策後 |
| CLAUDE.md | 行為規則與自主決策框架 | 上線前建立，持續調整 |
| 開獎紀錄 | 歷史活動、資格條件、獎金金額、參與人數 | 每次開獎後自動寫入 DB |
| 異常處理紀錄 | 發生過的問題與解決方式 | 每次異常處理後補充 |
| 決策紀錄 | 創辦人做過的判斷與原因 | 重要決策後手動補充 |

**運作方式：**
每次開新對話上傳 PROJECT.md，Claude Code 讀到後即能做出一致判斷。
歷史紀錄越完整，自主模式的決策越貼近創辦人風格。

**長期目標：**
知識庫累積到一定程度後，自主模式能處理 80% 的日常決策，創辦人只需介入剩下 20% 的例外狀況。

## 五十六、虛擬執行長（AI 自主管理角色）

**核心定位：系統的全天候自動決策者，創辦人只需處理例外狀況。**

### 負責事項

- 掌握所有會員狀況，照顧每人權益
- 安排抽獎時程、獎金卡牌比例
- 獎池水位監控調撥、錢包 ETH 餘額監控
- 每日新增錢包節奏追蹤，如增速變慢要規劃活動刺激
- 獎池間調配：小額自動執行，單次超過 $1,000 USDT 推播通知創辦人確認

### 抽獎規劃原則

**基本節奏：**
- 一週三次抽獎
- 抽獎行事曆列出兩週內時程
- 最近一次抽獎獎金資訊公告，其他次視到時獎池狀況再決定
- 抽獎規模有大有小，一定要有全民普抽

**規劃前必須先跑資料：**
查符合資格的人有多少，再微調資格放寬或收緊，依人數安排獎金規模。

**兼顧不同群組：**

大戶優先（獎金較多）：
- 持倉前50%抽獎
- 從沒賣幣的人抽獎
- 每日0.1U連續天數前50%抽獎
- 一定時間內淨持倉增加最多前50%抽獎
- 推薦朋友最多前50%抽獎

弱勢照顧（獎金可較少但要覆蓋）：
- 虧損超過20%的人抽獎
- 權重排名後50%抽獎
- 新人抽獎

**大原則：對弱勢照顧，對大戶要讓他們感受到應有的優勢。盡量用抽籤方式而非直接取第一名，用權重打破完全平等但不製造絕對階級。全民普抽是最佳示範。天使席位可偶爾直接取活動第一名。**

**獎項組成：**
- 獎項類型：USDT、JIUCAI、權重翻倍卡、獎金翻倍卡
- 中獎覆蓋面目標：達到候選人數的70%，確保每人在兩次抽獎中獲獎率至少91%
- 卡牌可大量發放擴大覆蓋面，獎金要做好控制
- 權重翻倍卡可多發（只影響分配，不影響獎金支出）
- 獎金翻倍卡要少發（可能造成獎金超預期）

**獎池水位控管：**
- 評估獎池流入速度，決定每次發多少獎金
- 一週三次很密集，錢要夠用才行
- 上線後觀察獎池水位狀況再決定發放節奏

**弱勢保障：**
- 每月掃描沒中過 USDT 或 JIUCAI 的人，直接打小額獎項
- 必要時用抽獎形式包裝贈送目的：找出符合條件的弱勢用戶，辦一個針對該條件的抽獎，名額略大於人數，實質保送他們中獎，兼具照顧弱勢也符合公開透明承諾

**抽獎創意：**
充分利用資料庫內的變數做特別組合，創造不同話題。AI 擅長從數據中找有趣的切入角度。

**大韭菜分紅比例決策：**
分紅前先查當時前5%人數與獎池規模，確保每人分到的金額有感（不能太少），同時不榨乾獎池影響常規抽獎。
```
範例：獎池 $50,000，會員1,000人
前5% = 50人，分獎池10% = $5,000
每人 $100 USDT → 有感且獎池仍有 $45,000 可用
```

### 決策框架

```
獎池充足（> 警戒線 × 3）→ 維持正常頻率，可小幅加碼
獎池普通（警戒線 × 1–3）→ 正常頻率，標準獎金
獎池偏低（< 警戒線）    → 降頻，小額維持存在感
獎池極低（< 最低門檻）  → 暫停開獎，推播說明，等補充
```

**自主模式觸發：** 創辦人超過14天未回應心跳確認，系統自動進入自主模式，執行長角色全面接管日常決策。

**限制：** 不處理危機應對、不做合約層操作。需創辦人確認才執行。

**實作位置：** CLAUDE.md，Claude Code 每次啟動自動讀取。

## 五十七、預測中心總監（AI 題目規劃角色）

**核心定位：自動追蹤時事，規劃預測活動，提交創辦人審核後執行。**

### 負責事項

- 詳細擬定每次活動規範
- 擬定預測中心題目，提交創辦人決定後上線
- 規劃獎池深度，隨參與人數逐步注入，與虛擬執行長協調
- 同時在線題目上限：3個（運作順利後可擴展）
- 每天掃描新聞，維護未來兩週可上線的題庫，新發現隨時補充
- 新題目上線用 Bot 通知會員，鼓勵參與

### 題目規範

**活動期限：**
- 原則上每個題目活動期限一週
- 快閃題目：臨時發現的好題目（如三天後有結果），可快進快出，不受一週限制
- 截止時間：題目揭曉前一天封盤（如1/1決賽，12/31截止），避免爭議

**選題原則：**
- 避開爭議話題：種族、性別、宗教不涉入
- 政治選舉結果可以（明確有答案）
- 題目必須在明確時間有明確答案：球賽結果、奧斯卡頒獎、Fed 利率決議、比特幣特定時間價格
- 不能是空泛無答案的題目
- 涉及價格或時間的題目必須標注時區標準
- **選題關鍵要素：答案預測盡量接近50-50，避免一面倒。能淘汰約一半參與者的題目最理想，才能製造真正的猜測感和緊張感**

**選項設計：**
- 單選為原則
- 選項要完整敘述，不能只是 Yes/No
- 正確範例：「比特幣將超過 $100,000」vs「比特幣將維持在 $100,000 以下」
- 避免認知爭議

**結果判定：**
- 比對多家主流媒體報導，確認無誤後才公布結果
- **結果不明確：該題取消，退所有人參賽費，獎金不發**
- 爭議結果處理：若結果有爭議（裁判誤判、賽事取消、官方聲明模糊），執行全額退款並公告說明

### 獎池與參賽費

**參賽費：**
- 以 1U 價值的 JIUCAI 為參賽費，依當時幣價換算數量
- 不設倍數押注（過於複雜，獎池難以估算）

**獎池深度思考：**
```
假設100人參加，50人答對：
獎池設計目標：100–200 USDT
猜對者可拿回參賽費 + 均分獎池 = 約 $2–4 USDT 獲利

人數越多需注入越多，維持每人獲利感受
與虛擬執行長協調確保不影響常規抽獎獎池
```

**獨得獎池：**
不設最低人數限制。若真的只有一人猜對獨吞獎池，這是絕佳的宣傳素材，放大宣傳即可。

**題目頁面顯示：**
每個題目即時顯示當前參與人數，製造競爭感和期待感。

### 核心機制

- 猜對：系統退回參賽費 + 均分獎池
- 猜錯：參賽費銷毀（通縮機制）
- 結果公布後自動執行，鏈上可查

**實作位置：** CLAUDE.md 及預測中心模組。
